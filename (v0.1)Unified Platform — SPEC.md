1. Overview

The Unified Platform is a core operating layer designed to support multiple “skins” (domain‑specific worlds) that share the same infrastructure but present different vocabularies, UI surfaces, and workflows.

The platform provides:

• Identity & Access Control
• Tenant Isolation
• Module Runtime
• Vault & Asset Storage
• Safety & Compliance (Guardian Core)
• Routing & Context Switching
• Telemetry & Observability


The three primary skins built on this platform are:

• University Skin — learning environments, classrooms, labs
• Legacy Game Studios Skin — game pods, studios, kid‑safe catalog
• Space‑LEAF / Space Car Skin — missions, vehicles, simulations


Each skin is a configuration layer, not a separate system.

---

2. Core Architecture

2.1 Identity Service

The Identity Service provides unified authentication and authorization across all tenants.

Capabilities

• Single Sign‑On (SSO) across all skins
• OAuth2 / OIDC support
• Role‑based access control (RBAC)
• Tenant‑scoped permissions
• Session management + refresh tokens


Core Entities

• User
• Role
• Permission
• TenantMembership


---

2.2 Vault & Storage

The Vault is the platform’s secure, versioned storage layer.

Responsibilities

• Store assets (images, audio, models, documents)
• Store build artifacts (games, simulations, course modules)
• Maintain immutable version history
• Enforce access rules based on tenant + role
• Provide signed URLs for time‑limited access


Data Types

• Asset
• Build
• Config
• Metadata


---

2.3 Routing Layer

The Routing Layer determines where a user goes and what context they operate in.

Responsibilities

• Route users to modules based on tenant + role
• Maintain context (classroom, game pod, mission)
• Enforce tenant boundaries
• Provide deep‑linking into modules


Context Object

{
  userId,
  tenantId,
  moduleId,
  role,
  contextType,   // classroom | gamePod | mission
  contextRef     // courseId | gameId | vehicleId
}


---

2.4 Safety & Compliance (Guardian Core)

Guardian Core is the platform’s kid‑safe and content‑safe enforcement engine.

Capabilities

• Text scanning (profanity, grooming, hate, violence)
• Media scanning (unsafe imagery, metadata flags)
• Build scanning (tags, assets, runtime behavior)
• Policy engine (age gates, region rules, parental controls)
• Runtime enforcement (kill‑switch, quarantine)
• Audit logging


Outputs

• SafetyReport
• PolicyDecision
• AuditEvent


---

2.5 Telemetry

Event Types

Dashboards

4. Module System

4.1 Module Contract

All modules must implement the following contract:

Inputs

• Identity context
• Tenant context
• Module context
• Permissions


Capabilities

• Read/write to Vault (within permissions)
• Emit telemetry events
• Request safety checks
• Access tenant configuration


UI Modes

• Embedded frame
• Fullscreen app
• API‑only service


---

4.2 Module Types

• Internal Modules — built by platform owners
• Tenant Modules — built by skin owners
• Third‑Party Modules — built by external studios/partners
• Sandboxed Modules — isolated execution with restricted APIs


---

5. External Studio / Third‑Party Integration

5.1 Onboarding Flow

1. Studio registers
2. Admin approves
3. Studio receives API keys / OAuth client
4. Studio gains access to submission portal


---

5.2 Submission Pipeline

Steps

1. Upload build
2. Extract metadata
3. Run automated safety checks
4. Run automated performance tests
5. Human review (if flagged)
6. Approve or reject
7. Publish to catalog


Artifacts

• Build
• SafetyReport
• PerformanceReport
• ReviewDecision


---

5.3 Runtime Sandbox

All third‑party games run in a sandbox with:

• Limited API surface
• No direct file system access
• No cross‑tenant communication
• Guardian‑enforced policies
• Kill‑switch on violation
